1. Stack & arborescence

Odoo 17 · Python 3.12 · PostgreSQL · OWL 2 · XML · JavaScript (ES Modules). Aucune dépendance JS tierce.

refuge-des-aventuriers/
├─ docs/                         # MCD, archi, guides (HTML), cahier des charges
├─ AI_LOG.md                     # journal IA (livrable 5)
├─ README.md
└─ odoo17-iut/
   ├─ docker-compose.yml         # services odoo / db / mailhog
   └─ odoo/local-src/            # ADDONS DU PROJET
      ├─ refuge_aventuriers/     # socle métier (produits, BoM, fidélité, POS, RH)
      ├─ refuge_table_order/     # apps OWL : client (QR), barman, clients, QR admin
      └─ refuge_planning/        # app OWL : planning + dispos

Chaque module suit la structure Odoo standard :

DossierContenu
models/Modèles Python (champs, contraintes, logique métier).
controllers/Routes HTTP (pages) et JSON-RPC (API des apps OWL).
static/src/Composants OWL (.js + .xml + .css), bundlés via le manifest.
data/Données XML (employés, contrats, dispos par défaut, setup POS).
security/Groupes & droits d'accès.
tests/Tests unitaires (--test-tags).

2. Monter l'environnement de dev

Docker Compose v2 n'est pas installé : utiliser docker-compose (v1). L'image nginx du compose est morte — ne lancer que odoo db mailhog.
cd odoo17-iut
docker-compose up -d odoo db mailhog

Odoohttp://localhost:32776 — DB refuge_aventuriers

Comptes : gérant admin/admin ; 4 employés (login = e-mail de travail, mot de passe refuge).

3. Appliquer une modification — procédure clé

3.a — Python (modèles / contrôleurs) ou XML (data / vues / sécurité)

# upgrade du / des modules concernés
docker exec odoo17-iut_odoo_1 /odoo/.venv/bin/odoo \
  -d refuge_aventuriers -u refuge_planning --stop-after-init --no-http
docker-compose restart odoo

Une modif contrôleur seul (sans nouveau champ/route) ne nécessite qu'un docker-compose restart odoo.

3.b — Assets OWL (JS / XML / CSS)

Un rafraîchissement navigateur NE suffit PAS, et un restart non plus : les bundles sont mis en cache en base comme ir.attachment nommés refuge_*.assets_%. Il faut les supprimer — ils se régénèrent au prochain chargement de page.
echo "env['ir.attachment'].sudo().search([('name','like','refuge_planning.assets_%')]).unlink(); env.cr.commit()" \
  | docker exec -i odoo17-iut_odoo_1 /odoo/.venv/bin/odoo shell -d refuge_aventuriers --no-http

Vérifier qu'une modif est bien servie en récupérant l'URL /web/assets/<hash>/refuge_*.assets_*.min.js depuis le HTML de la page (pas seulement par le nom de l'ir.attachment).

4. Pièges connus (à lire avant de coder)

PiègeConséquence & parade
Mots de passe staff effacés par -u Le hook post_init_refuge_staff ne tourne qu'à l'installation, pas au -u. Après tout -u touchant refuge_aventuriers, ré-affecter en odoo shell : pour chaque login staff u.password='refuge'.
Installation hr_contract à froid Crash column res_company.<X> does not exist. Pré-créer les colonnes en SQL (contract_expiration_notice_period, work_permit_expiration_notice_period) avant l'install.
Dépendance manifest Ajouter un module aux depends rend TOUS les modules refuge ininstallables tant qu'il n'est pas installé.
resource.calendar en XML Sans attendance_ids inline, Odoo garde son 40h Lun–Ven par défaut. Définir attendance_ids avec un (5,0,0) en tête.
24h vs 20h Le sujet dit 20h/sem ; le code utilise 24h (4 × 20 = 80h < 90h nécessaires pour couvrir 10h–01h sur 6 jours). Choix assumé — ne pas « corriger » sans en discuter.

5. Conventions OWL

  • Apps standalone : chaque page (planning_page, pages table_order) charge un bundle dédié refuge_*.assets_app via t-call-assets ; les fichiers sont inclus par glob static/src/**/* dans le manifest.
  • Communication : wrapper rpc.js maison basé sur fetch (JSON-RPC 2.0), dupliqué volontairement par module pour rester indépendant. Toujours gérer l'erreur réseau et la remonter à l'utilisateur.
  • État : useState ; un composant racine par app, sous-composants par responsabilité ; AppHeader partagé (thème refuge_aventuriers).
  • Temps réel : setInterval dans onMounted, toujours clearInterval dans onWillUnmount.
  • Permissions : vérifiées côté contrôleur (ex. _is_admin, _check_can_edit_employee) — le front masque, le back tranche.
  • Cohabitation POS : écrire dans pos.order (champ refuge_source), jamais un modèle de commande parallèle.
