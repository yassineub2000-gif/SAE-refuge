Le Refuge des Aventuriers

Schéma d'architecture — composants OWL & flux de données — Livrable 3

Les deux applications OWL (modules refuge_table_order et refuge_planning) sont des pages standalone servies par des contrôleurs Odoo. Chaque app monte une arborescence de composants OWL et communique avec le backend uniquement en JSON-RPC, via un mince wrapper fetch maison — aucune bibliothèque tierce. L'état est géré par useState (store réactif OWL).

4. Flux — Espace barman temps réel (polling ≤ 5 s)

Aucune action du barman n'est requise pour voir une nouvelle commande : le BarmanApp interroge le backend toutes les 5 s (cahier des charges : ≤ 10 s). Le timer est nettoyé dans onWillUnmount.

sequenceDiagram
    autonumber
    participant CL as Client (ClientApp)
    participant API as Controleur Odoo
    participant DB as pos.order (ORM)
    participant BA as BarmanApp

    CL->>API: POST /api/table/{token}/submit
    API->>DB: create pos.order (source=table, statut=en attente)
    Note over BA: boucle setInterval (5 s)
    loop toutes les 5 s
        BA->>API: POST /api/barman/orders
        API->>DB: search (en attente / en preparation)
        DB-->>API: commandes
        API-->>BA: liste JSON
    end
    BA->>API: set_status "en preparation" puis "prete"
    API->>DB: write refuge_kitchen_status
    Note over API,DB: a l'encaissement POS :
deduction stock (BoM) + credit fidelite

6.a — Principes d'architecture (grille d'évaluation OWL)

CritèreMise en œuvre
Découpage en composants1 composant racine par app + sous-composants par responsabilité (écrans client, OrderCard, WeekCalendar, AvailabilityHeatmap…). AppHeader mutualisé.
Gestion d'étatuseState (store réactif OWL) ; le ClientApp est une machine à états (state.screen).
Temps réelsetInterval 5 s dans BarmanApp (≤ 10 s exigé), nettoyé en onWillUnmount. OrderCard a son propre timer d'affichage.
CommunicationWrapper rpc.js basé sur fetch (JSON-RPC 2.0) ; gestion d'erreur réseau remontée à l'utilisateur ; aucune dépendance JS tierce.
Cohabitation POSL'app client écrit dans pos.order (champ refuge_source), pas de modèle parallèle.
Chargement des bundles. Chaque page (planning_page, pages table_order) charge un bundle d'assets dédié (refuge_*.assets_app) via t-call-assets, indépendant du backend web d'Odoo — d'où des apps OWL légères et autonomes.
